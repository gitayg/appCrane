Capabilities

Everything you need
to ship.

A full deployment platform on your own machine. No monthly fees, no vendor lock-in, no cloud bill surprises.

Multi-app on one server
Run as many apps as your server can handle. Each gets isolated PM2 processes, its own ports, and automatic Caddy routing.
Dual environments
Every app gets a production and sandbox environment, always-on. Test on sandbox, promote to production in one command.
Auto-HTTPS via Caddy
Caddy handles TLS certificates automatically via Let's Encrypt. Apps are reachable at your-domain.com/appname with no manual cert work.
GitHub webhook auto-deploy
Push to main and your sandbox deploys automatically. HMAC-verified webhooks keep the pipeline secure. Register directly from the dashboard.
Rollback in seconds
Symlink-based releases keep the last 5 builds. Rolling back is a symlink swap — instant, no re-build required.
Zero-downtime deploys
Start new → health check → swap → drain old. Production stays up throughout the deploy cycle.
Encrypted env vars
AES-256-GCM encrypted at rest. Admin is blocked from reading them by design — only app users can access their own secrets.
Health checks + alerts
Pings your /api/health endpoint every 30s. Auto-restarts on failure. Sends email when apps go down or recover.
AI-agent friendly API
Every operation has a curl equivalent. Point your AI agent to /agent-guide for a complete markdown doc with copy-paste commands.
Workflow

Deploy in
five steps.

From fresh Ubuntu server to running app in under ten minutes.

01
Install
Run the one-liner. Installs Node 20, Caddy, PM2, AppCrane, and creates an admin key.
02
Create app
crane app create with a slug, domain, and GitHub URL. AppCrane allocates ports, creates PM2 processes, and wires up Caddy routing.
03
Set env vars
crane env set or the dashboard. Encrypted at rest. Separate values per environment. Admin can't read them.
04
Deploy to sandbox
crane deploy --env sandbox. AppCrane pulls, builds, starts, health-checks, and swaps the symlink. Production is untouched.
05
Promote
crane promote. Copies code from sandbox to production with zero downtime. Never bypasses the sandbox step.
crane — deploy workflow
# Create the app
$ crane app create --name MyApp --slug myapp --repo https://github.com/org/myapp
✓ App myapp created. Available at https://your-domain.com/myapp
# Set env vars (encrypted, sandbox only)
$ crane env set myapp --env sandbox DATABASE_URL=postgres://... API_KEY=sk-xxx
✓ 2 vars set for myapp/sandbox
# Deploy to sandbox, check health, promote
$ crane deploy myapp --env sandbox
✓ Deploy #12 live — v1.4.2 — 23s
$ crane promote myapp
✓ myapp/sandbox → production (zero downtime)
# Something wrong? Rollback takes 1s
$ crane rollback myapp --env production
✓ Rolled back to v1.4.1
Security model

Admin can't read
your secrets.

Two distinct roles with a hard wall between infrastructure control and application data. Admin manages the platform. App users own their deployments.

Admin Platform operator — manages infrastructure
  • Create and delete apps
  • Create users and assign them to apps
  • View server health (CPU, RAM, disk)
  • Configure branding and settings
  • Cannot read env vars or /data/
  • Cannot deploy, rollback, or promote
  • Cannot view backups or configure health
Admin isolation is enforced at middleware level — not by convention. Even with direct database access, env vars are AES-256-GCM encrypted with a key only on-disk.
App User Developer — owns their app's lifecycle
  • Deploy, rollback, and promote (own apps only)
  • Read and write env vars (own apps only)
  • Configure health checks and webhooks
  • Create and restore backups
  • View logs and audit trail
  • Access /data/ directory
  • Cannot access other users' apps
